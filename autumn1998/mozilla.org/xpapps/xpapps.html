Introduction

This document describes some of the architectural and design ideas that are being considered for the Cross-Platform Applications module aka XPApps by the xpapps team at Netscape. The ideas proposed here are very priliminary and are not final designs carved on stone. This is a work in progress.

Objective

The objective of XPApps in the NGLayout based communicator is
¤ Develop internet applications like the Browser, Composer,
Mail & News reader based on the new layout engine, NGLayout.
¤ Develop an Application Shell that will be used to implement
the application suite mentioned above.
The Application Shell and the suite of applications must be written in a modular and Cross-Platform way leveraging the underlying layout engine and the services provided by the Core FrontEnd Toolkits module aka XPToolkits

Application Suite

This Section describes the objective #1. The Application suite includes the usual "big 3" listed above and a bunch of satellite applications that are required for the complete functioning of these "big 3" applications. These satellite applications include:
¤  User Profile Manager
¤  Preferences UI
¤  Security Manager
¤  Bookmarks
¤  History
¤  Address Book Manager
¤  Side Bar
¤  Calendar
¤  Chat
¤  Progress Indicator 		  ¤  Find
¤  Page Source
¤  Page Info
¤  About
¤  Open Page/URL
¤  File Open/Save
¤  FTP
¤  Privacy
¤  Cookies
¤  Password
All the applications will leverage the services offered by the App Shell

Application Shell

The Application Shell will encompass the basic and essential services required for the applications to bootstrap and operate. The term "Application Shell" merely indicates a module that provides some services to the apps and should not confused with the unix shell or a X-Window toplevel/Application shell. Some of the services that the AppShell will provide includes:
¤  Command line processing
¤  Window manager interactions
¤  Processing application resources
¤  Processing UI Specs from RDF/XML files
¤  Launching other applications based on a user action
¤  Interactions with other applications
¤  Synchronizing multiple instances of the application
¤  Creating the top-level window, (the four borders) with WM decorations
The Application Shell can also be manager of a collection of core services like libpref, libnet etc... which we will discuss in the next section

Application Service Provider

Bootstrapping and running an application involves initialising a bunch of services provided by the underlying modules like the core Layout engine and XPToolkits. The services and the order in which these services are initialized varies among applications. So there is a need for a module that identifies these requirements, dependencies and provide the appropritate services. An Application Service Provider (ASP) is the one that provides such services. Decisions about the service requirements and dependencies of the application on hand; can be either programmed or read in from a XML file. Reading from a file provides oppurtunities for building a application service provider that is independant of commmunicator and therefore can be used elsewhere.

Open Issues
¤  Interactions with the XPToolkits module.
¤  Ownership and management of the Message pump.
¤  Making the App Shell do the work of the App Service Provider
¤  Should the top-level window be platform specific or independant
These issues will be resolved in future design meetings.
