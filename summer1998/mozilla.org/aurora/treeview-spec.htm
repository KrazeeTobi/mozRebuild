Aurora Tree views and functionalities

Radha Kulkarni, Netscape communications corp,
Created: 27 July, 1998
Updated: 4 Aug, 1998

Objective:

This document details the general look and behavior of the tree views that will be used in Aurora work spaces. Specific workspace behaviors are detailed in Mike's Aurora Workspace Spec.

Tree View UI:

The tree view can be displayed in 3 modes
  • Popup - Tree view appears as a popup as soon as a folder in the Info bar is clicked. For info on the Toolbars look at David's Configurable Chrome Spec.
  • Docked - Tree view is embedded to the left side of the browser window.
  • Standalone - Tree view appears in an independent window. The tree view is in management mode when displayed in standalone mode
Each tree view has a tree control area, a title bar, and a tree structure.  The tree control area contains buttons that perform management operations on the tree view. The title bar displays the current workspace name and looks seamlessly blend in with the tree structure. The tree structure displays the hierarchical rdf data. The tree view has 2 behavioral states, navigation and management.
 

Navigation state:

The navigation state of the tree view provides quick and easy access to information and very minimal management capability. The behavior and look of the tree view in this  state is similar to the "Bookmarks" in the current 4.x product.  The tree view when invoked in the popup and docked mode by default behaves in navigation state. At this state, the tree structure that represents the rdf data by default does not display column headers. By default, only one column is displayed. There are no connecting lines between the rdf containers and URL nodes. URLs can be loaded in the browser view by single clicking on them. There are no context menus and no in-line editing  in this mode. URLs can be dropped into the tree view but not dragged out. Selection is not supported in this mode, ie., no visual feedback will be given when a node is selected. This is done to avoid any confusions in the behavior of the tree. Selections hint D&D, and availability of context menus which are not provided in this state. So, though a node will be "technically" selected when the user places the mouse over it, a visual "selected" feedback will not be given to avoid above described confusions. The primary operation supported in this state is single click of a URL node to load it in the browser window.

In this state, the tree control area contains 2 buttons.

  • See details:  Clicking this button displays the column headers of the tree structure there by enabling the user to perform few management operations like sorting etc...
  • close: This button is primarily useful in the docked mode. Clicking on this button will close the docked tree view.
Management state:
The management state allows advanced manipulation of information. The tree view when displayed in standalone mode by default appears in management state. This state can be compared to the "Edit Bookmarks" feature in the 4.x product. The tree structure is displayed with column headers. Folders and URLs in the rdf tree structure appear connected by a line.  RDF nodes can be selected, ie., a visual feedback of selection will be provided, nodes can be added or moved through Drag & Drop. Context menus are provided for all management operations. Inline editing is permitted.  A URL can be launched in the browser by double clicking on them.

Mode switches:

A tree view can be switched from popup, docked and standalone mode in the following ways.

1) A popup tree view can be docked to the browser window by using the tear-off feature of the popup menus.
2) A docked tree view can be made stand alone by dragging it out of the browser view.
3) The title bar of the tree view will provide context menu options which can switch the tree view from popup to docked or standalone mode.

It appears that windows will implement all the above 3 options.  Mac and  Xfe may have to resort to 3) option for mode switching operations. The 3) option will not be available in standalone mode. It s'd be noted at this time that tree-view modes are shared by all workspaces ie., when bookmarks are displayed in standalone mode, clicking on history or sitemap will bring up these workspaces in standalone mode. When the bookmarks are displayed in docked mode, clicking on history will replace bookmarks in the docked view.

It is not clear at this time, if rdf data states will be maintained across views. For example, if I sort the bookmarks when it is docked to the browser view and later bring it in up in popup or standalone, or switch to standalone mode, s'd the bookmarks appear sorted or not?
