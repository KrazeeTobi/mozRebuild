Mozilla Plug-in API Guide

This guide documents the Plug-in Application Programming Interface (API) for the Mozilla codebase. The information here applies to all browsers derived from the Mozilla codebase, including the Mozilla browser, Netscape Navigator 5, and Neoplanet.

Mozilla-based browsers provide an impressive set of capabilities for accessing and displaying information from the World Wide Web or company intranets. Plug-ins extend the browser to include a wide range of interactive and multimedia capabilities, while blending seamlessly into the interface.

The Mozilla Plug-in API Guide includes the information you need to write a plug-in for Mozilla-based browsers. It describes the new APIs and the steps for creating and building a plug-in for Mozilla-based browsers and for upgrading an existing plug-in for optimal performance on Mozilla-based browsers. It also describes how to design your new plug-ins so they will work in older versions of the browser. By following the guidelines in this document, plug-in developers can create a single plug-in binary that performs optimally in the newest version of the browser while maintaining full backward compatibility for use in older browsers.

About This Guide

The Mozilla Plug-in API Guide is designed for C++ developers who want to extend the capabilities of their Mozilla-based browser. This includes all developers who are writing new plug-ins for Mozilla-based browsers and all developers who have already released plug-ins for previous versions of the browser.

This is the guide to read if you want to write C++ code for plug-ins for Navigator 5.x or other Mozilla-based browsers. This guide provides reference information and instructions for creating a plug-in for all Mozilla-based browsers.

If you are new to plug-in development, please read the Plug-in Guide (http://developer.netscape.com/docs/manuals/communicator/plugin/contents.htm) before reading this document. Although the Plug-in Guide focuses on the development of plug-ins for 4.x versions of Navigator, much of the information in it is still valid.

Benefits of Using the New Plug-in API

Mozilla is a complete rewrite of previous versions of Netscape's browser. Using the Mozilla source code as its foundation, the new browser is designed to be small, fast, and standards-compliant. The new plug-in architecture offers many of advantages over the old architecture, including greater extensibility and modularity. Designing the new browser from the ground up has provided the opportunity to enhance the Mozilla plug-in API in a number of ways:
  • Plug-ins that offer LiveConnect APIs no longer depend on the Navigator 4 Java Virtual Machine (JVM). This means that end-users do not need to wait for the JVM to start before a plug-in runs, and consequently, performance is improved.
  • New START and STOP methods are provided that enable plug-ins to stop execution easily when the user leaves the page. This improves performance when the user returns to the page. The plug-in remains in memory so it resumes execution immediately when the user returns to it.
  • A plug-in can now ask the browser to change its size dynamically at run time, allowing a multitude of special effects.
Because of these and other improvements, all plug-in vendors are encouraged to upgrade their plug-ins to take advantage of these enhancements.

Upgrading Plug-ins Based on LiveConnect

If your plug-in provides a JavaScript-accessible LiveConnect API, then you must upgrade your plug-in to the new Mozilla plug-in API. LiveConnect, as used in Navigator versions 3 and 4, depended on the presence of the Netscape JVM. Mozilla supports Java through the Open Java Interface (OJI), giving users the freedom to use any OJI-compliant JVM as soon as it is released, and the Mozilla plug-in API has been redesigned to eliminate the dependency on the JVM. Existing plug-in binaries on Windows and Macintosh that feature a LiveConnect API will still function within the new browser, but calls to the LiveConnect API itself will fail silently until the plug-in has been upgraded to the Mozilla plug-in API. Upgrading a plug-in is a straightforward process that involves mapping the old API to the new one, providing a backward adapter file, and recompiling the plug-in to reflect the changes. Step-by-step instructions for upgrading plug-ins can be found later in this guide.

Backward Compatibility of Upgraded Plug-ins

Plug-ins upgraded to take advantage of these enhancements for Mozilla will remain backwardly compatible (on Windows and Macintosh) with previous browsers, so vendors will be able to release a single binary that runs in all browsers and provides the best possible user experience for the new browser. The performance gains will in turn make content and application developers even more likely to adopt plug-ins, benefiting users and plug-in vendors alike.

Overview of the New Plug-in Architecture

The new architecture includes several abstract base classes. All NPP functions, those implemented by the plug-in developer, become methods on the NPIPlugin, NPIPluginInstance, and NPIPluginStream classes. All NPN functions, those implemented by Mozilla, become methods of the NPIPluginManager, NPIPluginManagerStream, NPIPluginInstancePeer, and NPIPluginStreamPeer classes.

When a plug-in is loaded by the browser, an instance of the service manager is passed to the plug-in's initial entry point, NSGetFactory. The service manager can be used to access other browser services such as the memory allocator. The caller uses this plug-in object to get general information about the plug-in, such as the MIME type the plug-in handles or the name and description displayed in the About Plug-in page.

When an EMBED or OBJECT tag appears on the page, the NewInstance method is called on the plug-in object. It is the responsibility of this method to instantiate a subclass of NPIPluginInstance. The NewInstance method is passed an instance of NPIPluginInstancePeer and the newly created plug-in instance may use it to query the plug-in about its relationship to the page. The plug-in instance stores the plug-in instance peer in an instance variable for use by other methods.

After the plug-in instance is created, the plug-in is notified that an initial data stream is going to be fed to it through a call to nsIpluginInstance->NewStream. This only occurs if the OBJECT tag has a DATA attribute or the EMBED tag has an SRC attribute. The call to NewStream expects nsIPluginStreamListener to be returned to the browser so it can feed the stream to the plug-in. Subsequent calls to GetURL or PostURL already take parameters for the nsIPluginStreamListener, so NewStream is not called in those cases.

The nsIPluginStreamListener methods are called as the stream data comes in. OnStartBinding indicates that the stream is ready. Calls to OnDataAvailable give data to the plug-in and OnStopBinding indicates that the stream is finished. OnFileAvailable is only used when the call to GetStreamType returns either nSPluginStreamType_AsFile or nsPluginStreamType_AsFileOnly.

Instructions for Upgrading an Old Plug-in

Once you decide what you want your new plug-in to do, upgrading it to the new plug-in API involves these steps:
  1. Decide on a MIME type and file extension for the plug-in.
  2. Implement the three new C++ interfaces:
    • nsIPlugin
    • nsIPluginInstance
    • nsIPluginStreamListener
  3. Map all NPN* calls to the new C++ interfaces. NPN functions are the Netscape functions used with Communicator 4.x. See the Plug-in API Map later in this document for information on how to map methods from the Navigator 4 API to the new ones.
  4. Change all streaming methods to NSIPluginStreamListener. This allows you to implement different streaming methods for different MIME types so it is easier to modularize your programming code. On Linux only, upgrade the user interface from Motif to GTK. For more information, see Creating Plug-ins on Linux later in this document.
  5. Create ID or header files for the interfaces that you want to appear in JavaScript.
  6. Generate the .h and .xpt files for the plug-in.
  7. Ship the .xpt file with the plug-in.
  8. On Windows and Macintosh:
    1. Implement NPP_GetJavaClass in the backward adapter file (http://lxr.mozilla.org/mozilla/source/modules/plugin/bad/badapter.cpp).
    2. Ship the original .class file if you want your plug-in to work in Navigator 4.x with LiveConnect.
  9. Recompile.

NS_IMETHOD Initialize

Provides global initialization for a plug-in. Navigator calls this function only once: when a plug-in is loaded, before the first instance is created. This is the first function Navigator calls. Initialize tells the plug-in that Navigator has loaded it and provides global initialization. Allocate any memory or resources shared by all instances of your plug-in at this time.

After the last instance of the plug-in has been deleted, Navigator calls Shutdown. This releases allocated memory or resources.

Parameters
None

NS_IMETHOD Shutdown

Provides global deinitialization for a plug-in. Navigator calls this function once after the last instance of your plug-in is destroyed, before unloading the plug-in library itself. Use Shutdown to delete any data allocated in Initialize to be shared by all the instances of a plug-in.
Parameters
None

NS_IMETHOD SetWindow

Tells the plug-in when a window is created for it to draw into, or when its window is moved, sized, or destroyed. Navigator calls SetWindow after creating the instance to allow drawing to begin. Subsequent calls to SetWindow indicate changes in size and position; these calls pass the same nsPluginWindow object each time, but with different values. If the window handle is set to null, the window is destroyed. In this case, the plug-in must not perform any additional graphics operations on the window and should free any associated resources.

The data structure passed in SetWindow is an nsPluginWindow object, which contains the coordinates of the instance's area and various platform-specific data. This window is valid for the life of the instance, or until SetWindow is called again with a different value.

Parameters
The function has the following parameters: 
nsPluginWindow *window
 Contains the platform-specific drawing information and structures.

NS_IMETHOD GetURL

Requests the browser to create a stream for the specified URL. GetURL is used to load a URL into the current window or another target or stream. Plug-ins can use this capability to provide hyperlinks to other documents or to retrieve data from anywhere on the network. This is especially useful for enabling an existing application to operate on the web. For HTTP URLs, the browser resolves this method as the HTTP server method GET, which requests URL objects.

Make sure that the target matches the URL type sent to it. For example, a null target does not make sense for some URL types (such as mailto). The following recommendations about target choice apply to other methods that handle URLs as well.

If the target parameter refers to the window or frame containing the current plug-in instance, the instance is destroyed and the plug-in may be unloaded. If target is null, the application creates a new stream and delivers the data to the plug-in instance, through calls to nsIPluginStreamListener. This means that, if you want the plug-in to handle a new stream, no matter what the MIME type is, use null. If the application cannot locate the URL and retrieve the data, it does not create a stream for the instance.

When the plug-in instance is part of a regular browser window, and it uses a _blank target with a mailto or news URL, another blank browser window is opened along with the mail or news window.

When the plug-in uses a _self target, no other instance is created; the plug-in usually continues to operate successfully in its own window. The safest target is _blank, even though this creates an extra blank browser instance.

Parameters
The function has the following parameters: 
nsPluginManagerVariable variable
 Specifies the plug-in information the call gets. 
void* value
 Specifies the plug-in name, returned by the function.

NS_IMETHOD PostURL

PostURL works similarly to GetURL, but in reverse. GetURL reads data from the URL and either displays it in the target window or delivers it to the plug-in. PostURL writes data from a file or buffer to the URL and either displays the server's response in the target window or delivers it to the plug-in. If the target parameter is null, the new stream is passed to the plug-in regardless of MIME type. When you use PostURL to send data to the server, you can handle the response in several different ways by specifying different target parameters. If the target is null, the server response is sent back to the plug-in. You can get the data and save it in a file or use it in a program.

If you specify _current, _self, or _top, the response data is written to the same plug-in window and the plug-in is unloaded. If you specify _new or _blank, the response data is written to a new browser window. You can also write the response data to a frame by specifying the frame name as the target parameter. For HTTP URLs only, the browser resolves this method as the HTTP server method POST, which transmits data to the server. The data to post can be contained either in a local temporary file or a new memory buffer. To post to a temporary file, set the flag file to true, the buffer to the path name string for a file, and len to the length of the path string. The file-type URL prefix file:// is optional. If a file is posted with any protocol other than FTP, the file must be text with Unix-style line breaks ('\n' separators only). To post data from a memory buffer, set the flag file to false, the buffer to the data to post, and len to the length of buffer.

Possible URL types include HTTP (similar to an HTML form submission), mail (sending mail), news (posting a news article), and FTP (upload a file). Plug-ins can use this function to post form data to CGI scripts using HTTP or upload files to a remote server using FTP.

For protocols in which the headers must be distinguished from the body, such as HTTP, the buffer or file should contain the headers, followed by a blank line, then the body. If no custom headers are required, simply add a blank line ('\n') to the beginning of the file or buffer. PostURL is typically asynchronous: it returns immediately and only later handles the request.

Parameters
The function has the following parameters: 
nsISupports*pluginInst
 The current plug-in instance. 
const shar*url
 URL of the request specified by the plug-in. 
PRUint32 psotDataLen
 Length of the buffer. 
const char* psotData
 Path to the local temporary file or data buffer that contains the data to post. 
const char* target
 The destination where the URL will be displayed, a
window or frame.		 
nsIPluginStreamListener* streamListener
 The current stream.

NS_IMETHOD New Stream

Creates a new stream of data produced by the plug-in and consumed by the browser.

The MIME parameter is the MIME type of the plug-in to create. A plug-in can create another instance of itself by specifying its own MIME type and a new target name in a call to NewStream.

The stream is returned in the stream parameter. The plug-in can use this object in subsequent calls to nsIOutputStream Write to write data into the stream. When the plug-in has written all of its data into the stream, nsIOuputStream Close and nsIOutputStream Release terminates the stream and deallocates the Stream object.

The target parameter is the name of the target window or frame, or one of several special target names. For parameter values and information about how to use them, see GetURL.

NS_IMETHOD ShowStatus

You can use this function to make your plug-in user interface simulate the browser's behavior. When the user moves the cursor over a link in a browser window, the browser displays information about it in the status message area (on the lower edge of the browser window). If your plug-in has a button or other object that acts as a link when clicked, you can call Status to display a description or URL when the user moves the cursor over it. The browser always displays the last status line message it receives, regardless of the message source. Your message is always displayed, but you have no control over how long it stays in the status line before another message replaces it.
Parameters
The function has the following parameters: 
const char* message
 Specifies the string specified in the status bar of the plug-in's window.

Creating Plug-ins on Linux

Navigator 4.x for Linux was a Motif application that statically linked in and included in its distribution the Motif GUI binaries. As a result, plug-in vendors supporting Navigator 4.x on Linux used a Motif GUI for their plug-in binary.

On Linux, as on Windows and Macintosh platforms, the Mozilla Plug-in API in Mozilla and Navigator 5.x is still compatible with the plug-in API calls made by existing plug-in binaries. However, on Linux platforms, there is an additional change between Navigator 4.x and Moziila and Navigator 5.x.
In order to create a fully open source browser that can be compiled, used, and distributed by anyone without charge and without a dependency on software with a licensing fee, Mozilla and Navigator 5.x have switched from using the Motif GUI to the free and open source GTK GUI.

Differences exist between the Motif and GTK GUIs, and they cannot be used together. In particular, the API methods and the event handling systems are different. As a result, the Motif-based plug-in binaries cannot be used with a GTK-based browser application and existing plug-in binaries for Navigator 4.x for Linux will not run within Gecko and Mozilla-based browsers. This is unlike plug-ins on Windows and Macintosh platforms where existing plug-in binaries for Navigator 4.x run in Navigator 5.x.

External developers have verified that if an existing plug-in, using the Motif GUI and Navigator 4.x plug-in API, for Navigator 4.x for Linux is updated with a GTK GUI, the plug-in works in Navigator 5.x (with an updated GTK GUI with the same Navigator 4.x plug-in API calls). The same developers, however, also concluded that because of performance and other advantages of the Mozilla plug-in API, it is more efficient to update the plug-in from the Navigator 4.x plug-in API to the Mozilla API at the same time that the user interface is updated from Motif to GTK. Since all the existing Linux plug-ins must be updated from Motif to GTK and recompiled, using the Navigator 4.x plug-in API on Linux, even with an updated GTK GUI, will not be supported.

The compatibility and support for plug-in API versions and GUIs on Navigator for Linux is summarized in this table:

Motif GUI GTK GUI
Nav 4.x plug-in API Plug-ins using the Nav 4.x plug-in API and a Motif GUI are supported on Navigator 4.x for Linux. Plug-ins using the Nav 4.x plug-in API and a GTK GUI would be compatible (except for LiveConnect) with Navigator 4.x, but the use of the Navigator 4.x plug-in API (even with an updated GTK GUI) is unsupported on Navigator 5.x. Plug-in vendors should update their existing Linux plug-ins to the Mozilla Plug-in API and the GTK GUI in one step.
Mozilla plug-in API (This combination does not occur as Mozilla-based browsers on Linux will use a GTK GUI.) Plug-ins using the Mozilla Plug-in API and a GTK GUI are supported on Mozilla-based browsers such as Mozilla and Navigator 5.x for Linux.

Netscape recommends the following:

  • Vendors who have already released a plug-in for Navigator 4.x on Linux should immediately upgrade the plug-in to use both the GTK GUI and the Mozilla plug-in API.
  • All new development of plug-ins for Navigator on Linux should be done for the GTK GUI, Mozilla plug-in API, and Mozilla-based browsers, such as Navigator 5.x.
  • Vendors who need to support both Navigator 4.x and Navigator 5.x on Linux, should release two separate binaries:
    1. a plug-in for Navigator 4.x for Linux using Motif and Navigator 4.x plug-in API
    2. a plug-in for Navigator 5.x for Linux using GTK and the Mozilla plug-in API.

Note: On the Windows and Macintosh platforms, a single plug-in binary that has been upgraded to the Mozilla plug-in API can be run in Navigator 3.x, Navigator 4.x, and Navigator 5.x.
