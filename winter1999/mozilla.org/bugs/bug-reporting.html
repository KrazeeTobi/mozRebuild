how to write a good bug report
by Chris Yeh <cyeh@mozilla.org>

Here are some basic guidelines for writing a good bug report. Writing a bug report is the first step in getting a bug fixed. Writing a good bug report makes it easier for an engineer to find the bug. Finding the bug is at least half the work.

A good bug report will have the following characteristics. This document has instructions specific to mozilla, but it can be applied to almost any kind of software. Submitted bug reports should have at least some of this information in it.

  1. The software version.

    Each piece of software has a version number associated with it. This version number is used to identify when the software was produced as well as its feature and bug set. Mozilla uses a version number scheme of:

    major-revision.minor-revision<development-stage><development-number>-YYMMDDHH

    Where YY a two digit year, MM the month, DD the day, and HH the hour that the binary was built. <development-stage> is either "d" for development, "a" for alpha", "b" for beta. <development-number> represents the the number of that stage.

    So a build number of 5.0a1-98091501 reads "version 5.0 alpha 1, built in 1998 on the 9th month on the 15th day of the year at 1 am.

    You can find out the build number of your binary by typing about: in the location bar in a browser window.

  2. The runtime environment.

    Translated loosely: The kind of computer and platform are you using. This includes information such as:

    • Platform.
      The computing platform that you're using.

    • Operating system version.
      The version of the operating system for the platform. If it's a beta of the operating system, we need to know that.

    • Computer make and model.
      The manufacturer and model of your computer. Sometimes bugs only exist on a particular type of machine.

    • Installed memory.
      The amount of memory that your computer has installed.This is important since it may clue us in to crashes related to low memory conditions or memory corruption.

    • Other hardware information.
      Any other information about your hardware that you think may be related to the bug. If you have a screen redraw problem, mentioning the video card and monitor may be useful. Mentioning the kind of disk drive you have would not be helpful.

  3. What you were doing when you crashed.

    This may sound easy, but this is the hardest part about documenting a bug. Ironically, it's also the most important part of a bug report. When writing this part:

    • Give lots of detail.
      Write as much detail as possible, even if you think you're going overboard. Writing "I crashed when I closed a download window!" isn't very helpful. Writing "I crashed when I closed a download progress window while downloading "foo.tgz" file from web site "www.fooness.com" with 10% remaining." is a lot better.

    • Provide a URL if at all possible.
      If you crash while viewing a page, please please please give us the URL.

    • Random? Or consistent? Is there a pattern?
      If the bug shows any pattern at all, document the pattern. Even if it's nothing more than "When I view Kanji based web pages, I tend to crash soon after.", mention it. More information is better than less.

    • Distilling the bug and creating a recipe.
      If you can narrow down the bug and provide a short procedure on how to reproduce it, you increase the chance an engineer will be able to replicate it. If they can't replicate your bug, it becomes almost impossible for them to fix.

  4. Debugging information and asserts: the scene of the crime.

    If you have any kind of information generated by a debugger or by the program (via asserts) you should include it. However, remember the following:

    • Keep them short.
      Paging through reams of debugging logs doesn't do anything other than slow down the engineer and the bug tracking system. Just give enough information for context.
