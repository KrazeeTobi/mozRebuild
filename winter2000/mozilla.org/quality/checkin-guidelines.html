Checkin Guidelines for the Seamonkey Tree

These are general checkin guidelines for the source tree for the seamonkey project. Additional restrictions may be active during different phases of the project. Watch Tinderbox and the seamonkey project newsgroup for updates to this policy.

Must

  1. Build
    2. Build mozilla with change
  2. Precheckin tests
    3. Verify change passes Precheckin tests
  3. Test
    4. Run any change specific test
  4. Got Permission ?
    5. Get permission of module owner
  5. Ok to Checkin ?
    6. Check Tinderbox to ensure tree is open and green across platforms
  6. Checkin
    7. Checkin log should include a clear explanation of the fix. Include a bug number and reviewer (r=<email>) if there is one.
  7. Watch Tinderbox
    8. After checkin, Watch tinderbox for atleast 1 green cycle per platform; be reachable and ready to help. Should problems arise notify sheriffs and hook and fix the problem. If problem can't be fixed, back changes out.

Should

  1. Build on muliple platforms:
    2. Build mozilla with change on multiple platform. Get help from build buddies from other platforms.
  2. Code Reviews
    3. Get changes code reviewed. Code reviewers may use the Code Reviewers Guide.
  3. Carpools
    4. Landing of Branches and large checkins should be coordinated with the sheriffs and the build team.
  4. No Regressions
    5. Change should be an improvement and shouldn't cause regressions however small
If there are any specific questions or exceptions, contact sheriffs and hook immediately.
